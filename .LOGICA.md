# .LOGICA.md — Lógica interna do sistema

> Este arquivo descreve **como o sistema pensa e decide** — os fluxos,
> as máquinas de estado e as regras de decisão por trás de cada tela.
> Não repete o que já está no `README.md` (uso/instalação) nem no
> `PROMPT.md` (arquitetura/contexto para IA); aqui o foco é **o
> raciocínio passo a passo** de cada processo, para quem for alterar
> uma regra de negócio entender exatamente o que vai mudar e o que
> pode quebrar.
>
> **Regra do projeto: sempre que a lógica de negócio mudar, atualize
> este arquivo junto com `README.md` e `PROMPT.md`.** Os três precisam
> continuar contando a mesma história do sistema real.

---

## 1. Ciclo de vida de uma chamada (o fluxo central do sistema)

Este é o fluxo que todo o resto do sistema existe para suportar.

```
1. Um aluno é cadastrado (Admin) ou importado via CSV -> nasce com
   status = "aguardando", ativo = 1, vinculado a uma sala e a um
   ano letivo.

2. O aluno aparece na fila de 3 lugares, ao mesmo tempo, sempre que
   status = "aguardando" E ativo = 1 E não está marcado "faltante" hoje:
     - Kiosk da sala dele (/kiosk/<sala_id>)
     - Painel de Presença (/presenca/)
     - API (/api/chamadas)

3. Um operador clica "CHAMAR" no Kiosk (ou dispara via API/evento
   Socket.IO "chamar_aluno" com {aluno_id, guiche?}).

4. No servidor (services.chamar_aluno), dentro de UMA transação:
     a. Busca o aluno + nome da sala.
     b. Se o aluno não existe OU status != "aguardando" -> retorna None
        (protege contra duplo clique / duas telas chamando ao mesmo
        tempo: só a primeira chamada "ganha").
     c. UPDATE alunos SET status = 'chamado'.
     d. INSERT em `chamadas` (snapshot: nome, turma, sala_nome, foto
        no momento exato da chamada — não é um JOIN ao vivo, então o
        histórico nunca muda mesmo que o aluno seja editado/excluído
        depois).
     e. Retorna o dict da chamada criada.

5. Se retornou uma chamada válida:
     - Grava log de auditoria (tipo="chamada").
     - socketio.emit("aluno_chamado", chamada) SEM "room" -> vai para
       TODAS as telas conectadas ao servidor (Kiosk de todas as salas,
       todas as TVs, Presença, Admin).

6. Cada tela que recebe "aluno_chamado" decide, no PRÓPRIO JAVASCRIPT,
   se aquele evento é relevante para ela:
     - Kiosk da sala que chamou: remove o card da fila, adiciona em
       "Últimos chamados".
     - Kiosk de outras salas: ignora (o aluno não está na fila dele).
     - Presença: remove o card da fila (com uma pequena animação).
     - Screen dedicado a uma sala (`salaNome` != null): só reage se
       `chamada.sala_nome === CONFIG.salaNome`; senão ignora
       completamente (nem exibe, nem narra, nem entra na lista
       "Últimos chamados" daquela TV). É ASSIM que "o chamado não sai
       em todas as TVs" — não existe nenhuma segmentação no servidor,
       é 100% filtro client-side sobre um broadcast global.
     - Screen geral (`/screen/geral`, sem `salaNome`): reage a
       qualquer chamada.
     - Admin (dashboard): atualiza indicadores.

7. Se o aluno não compareceu, o operador clica "🔁 Repetir" (na lista
   "Últimos chamados" do Kiosk) ou envia "rechamar_aluno". Isso NÃO
   muda o status do aluno (ele já está "chamado") — só cria um novo
   registro em `chamadas` (tipo="rechamada") e dispara um novo
   broadcast "aluno_chamado", repetindo a exibição/narração na TV.
```

**Por que "broadcast + filtro no cliente" em vez de "rooms" do
Socket.IO?** Foi uma escolha deliberada de simplicidade: o servidor
não precisa saber quantas TVs existem, nem gerenciar inscrições em
salas — cada tela só olha para o campo `sala_nome` do evento que já
recebe de qualquer forma. Ao adicionar uma sala nova, nenhum código de
servidor muda; só aparece uma opção a mais na lista de seleção do
Kiosk/Screen.

## 2. Máquina de estados do aluno

Um aluno tem **três dimensões de estado independentes**, que juntas
decidem se ele aparece na fila:

| Campo | Valores | Quem altera | O que significa |
|---|---|---|---|
| `status` | `aguardando` \| `chamado` | `chamar_aluno()` (nunca volta sozinho para `aguardando`) | Se já foi chamado nesta "rodada" |
| `ativo` | `1` \| `0` | Admin/Supervisor (cadastro) ou Operador (Kiosk → Gerenciar alunos) | Se o aluno ainda estuda na instituição (0 = transferido para outra escola) |
| presença do dia (tabela `presencas`) | `presente` (implícito, sem registro) \| `faltante` (registro explícito) | Admin/Supervisor ou Operador | Se o aluno está na instituição HOJE |

**Regra de entrada na fila** (`listar_alunos(..., excluir_faltantes_hoje=True)`,
usada só pelo Kiosk):

```
na_fila = (status == "aguardando")
      AND (ativo == 1)
      AND (presença de hoje != "faltante")
```

Note que a **ausência** de um registro em `presencas` para o dia de
hoje é tratada como "presente" (`COALESCE(p_hoje.status, 'presente')`)
— ninguém precisa confirmar presença de todo mundo, todo dia; só é
preciso marcar quem faltou. Isso é resolvido com um `LEFT JOIN` com a
tabela `presencas` filtrado pela data de hoje (`date('now', 'localtime')`)
em toda consulta que precisa saber a presença do dia (`listar_alunos`,
`obter_aluno`).

**Voltar um aluno chamado para a fila:** hoje não existe uma ação
explícita "devolver para aguardando" — o fluxo esperado é sempre
"chamar" → "rechamar" se não vier. Se for necessário reabrir a fila
para um aluno (ex.: erro do operador), isso hoje só é possível editando
o registro diretamente ou recriando a lógica — **ponto de atenção para
quem for evoluir esse fluxo**.

## 3. Ano letivo e presença diária (Módulo 12)

- Um aluno é um cadastro **permanente**: ele nunca é duplicado nem
  recriado ao trocar de ano letivo — apenas `ano_letivo_id` aponta
  para o ano letivo vigente no momento do cadastro (ou o "ano letivo
  atual" configurado, se não informado).
- Existe sempre, no máximo, **um** "ano letivo atual" — guardado em
  `configuracoes` (chave `ano_letivo_atual_id`), definido pelo
  Admin/Supervisor em `/admin/anos-letivos` (botão "Definir como
  atual"). É o valor usado como padrão ao cadastrar um aluno novo
  sem informar `ano_letivo_id` explicitamente (`criar_aluno()` chama
  `obter_ano_letivo_atual_id()` quando o parâmetro vem `None`).
- `marcar_presenca()` é um **UPSERT** por `(aluno_id, data)`
  (`ON CONFLICT ... DO UPDATE`): marcar a mesma data duas vezes para o
  mesmo aluno atualiza o registro existente, nunca duplica.
- **Quem pode marcar presença/ativo/inativo:** qualquer perfil logado,
  através de `/kiosk/gestao/alunos` (rotas com `@login_required`, sem
  restrição de perfil) — é o trabalho do dia a dia do usuário padrão
  (operador). O cadastro completo do aluno (nome, turma, código, CPF,
  observações, prioridade, sala, ano letivo, foto "oficial") continua
  restrito a administrador/supervisor em `/admin/alunos/*`.

## 4. Importação CSV — não duplicar é a regra de ouro

**Alunos** (`aluno;serie;sala;foto`, só Admin/Supervisor):

```
para cada linha:
    se "aluno" vazio -> ignorado (contabilizado em resumo["ignorados"])
    se "sala" informada e não existe -> cria a sala automaticamente
                                         (resumo["salas_criadas"] += 1)
    procura aluno existente por (nome, turma) exatos
        -> se existe: ATUALIZA (não duplica), preservando foto/código/
           cpf/observações já cadastrados quando o CSV não os informa
        -> se não existe: CRIA um aluno novo
```

**Salas** (`nome;descricao;cor`, só Admin):

```
para cada linha:
    se "nome" vazio -> ignorado
    procura sala existente por nome (case-insensitive, sem espaços nas pontas)
        -> se existe: ATUALIZA
        -> se não existe: CRIA, com `ordem` = maior ordem atual + 1
           (entra no fim da lista de seleção do Kiosk/Screen)
```

A chave de deduplicação de aluno é **(nome, turma)**, não `codigo` —
então dois alunos homônimos na mesma turma vão colidir na importação
(o segundo "atualiza" o primeiro em vez de criar um novo). Se isso vier
a ser um problema real, o critério de deduplicação precisa mudar para
usar `codigo` (matrícula) quando disponível.

## 5. Permissões — como o sistema decide "pode ou não pode"

Três perfis, em ordem crescente de poder: `operador` < `supervisor` <
`administrador`. A checagem acontece em duas camadas complementares:

1. **`login_required`** (`routes/auth.py`): garante que existe uma
   sessão válida (`usuario_id` na sessão E o usuário ainda existe E
   está ativo no banco — ver seção 7, "sessão fantasma"). Não olha
   para o perfil, só se há alguém logado.
2. **`perfil_required(*perfis)`** e o `before_request` de
   `admin_bp` (`restringir_acesso`, em `routes/admin.py`): checam se
   `session["perfil"]` está entre os perfis autorizados para aquela
   rota específica. Rotas de Usuários/Configurações/Backup exigem
   estritamente `administrador` — nem `supervisor` entra.

**Não existe checagem de perfil no Kiosk nem no Screen** (rotas
públicas) além do que já foi descrito: as rotas de "gestão" dentro do
Kiosk (`/kiosk/gestao/*`) pedem login mas não pedem perfil específico
— qualquer perfil logado (inclusive `operador`) pode ativar/desativar
aluno, marcar presença e trocar fotos. O mesmo vale para o modal de
ações aberto a partir de um card na tela de Presença (Módulo 17): o
upload de foto reaproveita literalmente a mesma rota
`kiosk.gestao_aluno_foto`; só o link "Editar cadastro completo" é
condicional a `perfil_logado in ('administrador', 'supervisor')`,
decidido no servidor ao renderizar o template (`templates/presenca.html`),
não escondido só via CSS/JS — quem não é admin/supervisor nem recebe
o `href` daquele link no HTML.

## 6. Kiosk "modo simplificado" — uma tela, dois comportamentos

Controlado por uma única flag em `configuracoes`
(`kiosk_modo_simplificado`, ligado por padrão). O MESMO template
(`kiosk.html`) e a MESMA rota renderizam de forma diferente conforme
essa flag:

```
se kiosk_modo_simplificado == "1" (padrão):
    mostra:  logo, nome da instituição, botão "↩ Trocar sala"
             (só quando já dentro de uma fila), botão "🔐 Entrar"
    esconde: campo de guichê, atalhos (admin/gestão/presença/TV),
             toggle de tema, nome do usuário logado, botão "Sair"
se == "0" (desligado em Configurações):
    mostra tudo -- pensado para quando o próprio operador logado usa
    o Kiosk como estação de trabalho, não como terminal público mudo
```

O botão de login (Módulo 16) aparece **sempre** que não há ninguém
logado, independente do modo — é o único jeito de sair do modo
simplificado a partir do próprio terminal público.

## 7. Sessão e "sessão fantasma"

Uma sessão Flask guarda `usuario_id`, `usuario_nome`, `perfil`. O
problema: se a conta correspondente a esse `usuario_id` for excluída
(ou desativada) enquanto a sessão continua "válida" em outro navegador/
terminal, qualquer ação que tente **gravar** algo referenciando aquele
id como chave estrangeira (log de auditoria, chamada, presença) quebra
com `sqlite3.IntegrityError: FOREIGN KEY constraint failed`, porque o
SQLite recusa a gravação.

**Duas camadas de proteção, propositalmente redundantes:**

1. **Camada de dados** (`database/services.py`, helper
   `_id_usuario_valido(usuario_id)`): antes de qualquer `INSERT` que
   use `usuario_id`/`operador_id`/`registrado_por` como FK, verifica
   `SELECT 1 FROM usuarios WHERE id = ?`. Se não existir, grava `NULL`
   no lugar (a coluna é `... REFERENCES usuarios(id) ON DELETE SET
   NULL`, então isso é seguro) em vez de deixar o `INSERT` falhar.
   Cobre inclusive os eventos Socket.IO (`chamar_aluno`,
   `rechamar_aluno`), que não passam pelo `login_required` HTTP.
2. **Camada de sessão HTTP** (`login_required` em `routes/auth.py` e a
   rota `/` em `app.py`): em vez de só checar
   `"usuario_id" in session`, busca o usuário de verdade
   (`obter_usuario_por_id`) e confere `usuario["ativo"]`. Se a conta
   não existe mais ou está inativa, `session.clear()` + mensagem
   amigável ("Sua sessão não é mais válida...") + redireciona para
   login, em vez de deixar a sessão "zumbi" continuar até bater em
   algum erro técnico.

A camada 1 é a que realmente evita o crash (é onde o dado é gravado);
a camada 2 é a que dá uma experiência decente para quem está com a
sessão inválida, em vez de só evitar o erro silenciosamente.

## 8. Pareamento sala ↔ TV (Screen) — tudo no navegador, nada no servidor

O servidor **não sabe** qual TV física está em qual sala. Quem sabe
disso é o `localStorage` do PRÓPRIO NAVEGADOR de cada TV:

```
1ª vez que uma TV abre em /screen/:
    -> servidor renderiza modo="selecionar": grade com todas as
       salas ativas (GET /admin ou config, não importa a origem)
    -> a pessoa que está configurando aquela TV toca na sala correta
    -> JS (screen_selecionar.js) salva no localStorage:
           chamada_alunos_tv_sala_id = <id da sala escolhida>
    -> navega (redirect real de página, não SPA) para /screen/<sala_id>

próximas vezes que a MESMA TV (mesmo navegador) abre em /screen/:
    -> JS lê o localStorage, já encontra o id salvo
    -> redireciona sozinho para /screen/<sala_id>, sem mostrar a
       grade de novo

dentro de /screen/<sala_id>:
    -> servidor manda CONFIG_SCREEN.salaId / salaNome já preenchidos
    -> screen.js usa CONFIG.salaNome para filtrar o evento
       "aluno_chamado" (ver seção 1, passo 6)
    -> botão "🏫 Trocar sala" apaga a chave do localStorage e volta
       para a tela de seleção
```

Consequência prática: **o pareamento é por navegador, não por
dispositivo cadastrado no banco**. Se alguém limpar os dados do
navegador da TV, ou abrir `/screen/` em outro navegador/aba anônima
daquela mesma TV, ela volta a pedir a seleção de sala. Isso foi uma
escolha deliberada para não precisar de nenhum cadastro/autenticação
de dispositivo — o Admin também pode pular esse passo manual
configurando o link direto de cada TV para `/screen/<id-da-sala>`
(disponível em Salas → "Abrir TV" / QR "QR TV").

## 9. Destino da chamada — texto fixo, independente da sala

Desde o Módulo 15, o que é **narrado e exibido na TV** durante uma
chamada não é mais o nome da sala/TV que disparou a chamada — é
sempre um texto fixo e configurável
(`configuracoes.destino_chamada`, padrão **"Portaria de Saída"**),
injetado em todo template via `app.py -> inject_globals()` e também
como `CONFIG_SCREEN.destinoChamada` para o JavaScript. O `sala_nome`
da chamada continua existindo no payload do evento e é usado (a) para
o roteamento client-side (seção 1 e 8) e (b) para exibir de qual sala
veio no Kiosk/Presença/Admin — só não aparece mais como "para onde o
aluno deve ir".

## 10. Aviso de dados desatualizados (Módulo 11)

Sempre que Admin ou a "gestão" do Kiosk fazem um CRUD em `salas` ou
`alunos` (criar, editar, excluir, importar, trocar ativo/foto), a rota
chama `_notificar_dados_atualizados(tipo)`, que emite
`socketio.emit("dados_atualizados", {"tipo": tipo})` para todas as
telas. As telas que dependem de listas montadas no carregamento da
página (Kiosk, Screen no modo seleção, Presença) escutam esse evento e
mostram um aviso não-intrusivo ("dados atualizados, clique para
recarregar") em vez de recarregar sozinhas — para não interromper um
operador no meio de uma ação. É um mecanismo deliberadamente separado
do `aluno_chamado` (que É imediato e automático, porque uma chamada
precisa aparecer na hora).

## 11. Backup automático

Um greenlet do `eventlet` (`_iniciar_backup_automatico`, em `app.py`)
dorme `BACKUP_INTERVALO_HORAS` (padrão 24h) e chama `criar_backup()`
em loop infinito, desde o boot do processo — não depende de cron nem
de agendador externo. Falhas de backup são logadas
(`app.logger.exception`) mas nunca derrubam o servidor. Backups
manuais (tela Admin → Backup) chamam a mesma função `criar_backup()`
sob demanda. `BACKUP_MANTER_ULTIMOS` (padrão 30) limita quantos
arquivos ficam guardados — os mais antigos são apagados a cada novo
backup criado.

## 12. Onde cada tela busca o que mostra (resumo rápido)

| Tela | Carregamento inicial (HTTP/Jinja) | Atualização depois (Socket.IO) |
|---|---|---|
| Kiosk (fila de uma sala) | `listar_alunos(sala_id, status="aguardando", ativo=1, excluir_faltantes_hoje=True)` | `aluno_chamado` (remove card), `dados_atualizados` (aviso) |
| Screen (seleção) | `listar_salas(apenas_ativas=True)` | `dados_atualizados` (aviso) |
| Screen (sala/geral) | `listar_ultimas_chamadas_sala` (sidebar) | `aluno_chamado` (filtrado por `salaNome` quando aplicável) |
| Presença | `listar_alunos(status="aguardando")` (sem excluir faltantes/inativos — mostra o "estado real" da fila) | `aluno_chamado` (remove card), `dados_atualizados` (aviso) |
| Admin (dashboard) | consultas agregadas (contagens) | `aluno_chamado`, `dados_atualizados` (atualiza indicadores) |

Note a diferença sutil entre Kiosk e Presença: o Kiosk filtra
`ativo=1` e `excluir_faltantes_hoje=True` (é uma fila operacional —
não faz sentido oferecer para chamar quem está inativo/faltante); a
Presença hoje **não** aplica esses dois filtros — mostra
literalmente todos os "aguardando", incluindo quem está inativo ou
faltante no dia. **Ponto de atenção:** se isso não for intencional
para o caso de uso atual, é uma linha só (`ativo=1,
excluir_faltantes_hoje=True`) para alinhar o comportamento das duas
telas.

## Como manter este arquivo útil

Ao alterar uma regra de negócio (não um detalhe visual/de rota, mas
"o que o sistema decide e por quê"), pergunte: essa mudança altera
algum dos fluxos/tabelas acima? Se sim, atualize a seção
correspondente aqui, e confira se `README.md` (o "o quê", para
humanos) e `PROMPT.md` (o "contexto geral", para IA) também precisam
refletir a mudança.
